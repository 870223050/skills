# Push LuckMonster

## 概述

使用这个 skill 时，先把项目整理成适合提交的状态，再执行 Git 提交与推送。默认面向当前 Godot 项目工作目录，提交说明、README 文案和过程反馈优先使用中文。

## 工作流

1. 确认项目根目录。
2. 先补齐或更新 `README.md`。
3. 校验 Git 仓库状态与远端配置。
4. 整理本次提交内容，确保代码和美术资源都纳入检查范围。
5. 使用中文提交说明完成提交并推送。

## 第一步：确认项目根目录

- 优先在当前工作目录执行。
- 看到 `project.godot` 时，将该目录视为项目根目录。
- 如果当前目录没有 `project.godot`，再向上查找；仍找不到就停止，并说明无法确认 Godot 项目根目录。
- 不要假设这是一个已经初始化过的 Git 仓库，先检查再决定下一步。

## 第二步：更新 README.md

- 如果项目根目录下没有 `README.md`，就新建一个。
- 如果已有 `README.md`，先读取现有内容，再在保留有用信息的基础上补齐。
- 默认使用中文撰写，除非用户明确要求其他语言。
- README 至少覆盖这些内容：
  - 项目简介
  - 当前实现内容
  - 运行方式
  - 目录结构
  - 后续计划或待完善内容
- 可优先参考 [references/readme-template.md](references/readme-template.md) 的结构，但不要机械照抄；要根据实际项目内容改写。
- 如果项目内容还很早期，要明确写出“当前为原型/开发中状态”，不要伪造不存在的功能。

## 第三步：检查 Git 状态

- 先执行 `git status --short --branch`。
- 如果当前目录还不是 Git 仓库，执行初始化流程：
  - `git init`
  - `git branch -M main`
- 如果仓库已存在，只在必要时补充配置，不要覆盖用户已有历史。
- 不要使用破坏性命令，例如 `git reset --hard`、`git checkout --` 或强制清空工作区。
- 如果存在未跟踪文件、生成文件或引擎缓存，先结合 `.gitignore` 判断是否应纳入提交。

## 第四步：配置远端仓库

- 目标远端固定为 `https://github.com/870223050/luckMonster.git`。
- 如果没有 `origin`，添加：
  - `git remote add origin https://github.com/870223050/luckMonster.git`
- 如果已有 `origin` 且地址一致，继续后续步骤。
- 如果已有 `origin` 但地址不同，先停止并向用户说明差异，再决定是否修改远端；不要静默覆盖。

## 第五步：整理提交内容

- 提交前先再次检查 `git status`，确认本次要提交的文件范围。
- 代码和美术资源都需要纳入提交检查，不能只提交脚本或只提交资源。
- 优先检查这些项目内容：
  - `README.md`
  - `project.godot`
  - 逻辑脚本与场景文件
  - `Core/`
  - `Ex/`
  - `Res/`
  - `UI/`
  - 实际使用到的贴图、图标、音频、配置和导入文件
- 对自动生成且不应入库的目录保持谨慎，例如 `.godot/`；除非项目明确需要，否则不要默认提交。
- 如果发现美术资源引用了但文件未入库，要优先补齐，避免推送后项目打不开或素材丢失。
- 如果用户没有指定提交信息，就根据这次改动自动生成一条自然、准确的中文提交说明。
- 提交说明默认使用中文，保持“动作 + 内容”的短句形式，例如：
  - `初始化提交：补齐 Godot 项目骨架、README 与基础资源`
  - `功能提交：新增主界面、核心脚本与对应美术资源`
  - `整理提交：同步场景文件、贴图资源与项目说明`

## 第六步：提交与推送

- 首次提交的常见顺序：
  - `git add .`
  - `git commit -m "<中文提交说明>"`
  - `git push -u origin main`
- 如果已经有后续提交，则根据当前分支情况执行正常 `git push`。
- 如果推送失败且错误来自认证、权限或远端分支保护，停止自动重试，并明确告诉用户阻塞点。
- 如果没有任何可提交改动，不要创建空提交；直接汇报“工作区无新增改动”。

## 第七步：输出结果

- 完成后汇报这些信息：
  - README 是否已新建或更新
  - Git 是否为本次初始化
  - 当前分支名
  - 最新提交说明与提交哈希
  - `origin` 地址
  - 是否推送成功
- 如果中途因为远端冲突、认证失败或目录判断不清而停止，要明确说明停在第几步，以及下一步需要用户确认什么。

## 处理原则

- 默认用中文撰写提交说明、README 文案和过程说明。
- 先文档、后 Git：不要跳过 README 直接推送。
- 尊重已有改动：看到脏工作区时，理解并吸收已有变化，不随意回退。
- 遇到非显然风险时暂停，例如：
  - 已有 `origin` 指向别的仓库
  - 远端存在未知历史且本地不是其克隆
  - 推送会覆盖他人分支
- 需要暂停时，先汇报现状，再让用户决定。
