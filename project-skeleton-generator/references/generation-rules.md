# 生成规则

## 先整理输入

- 从需求文档抽出功能模块、页面清单、交互动作、状态流转和关键数据对象。
- 从需求文档抽出需要单独留接口的玩法系统，例如战斗、背包、任务、商店、存档或引导。
- 如果文档同时给出页面和流程，先按功能域分组，再决定目录和文件落点。

## 再决定骨架策略

- 如果仓库已有目录和命名约定，优先继承现有约定。
- 如果仓库为空但文档已经明确引擎或框架，沿用文档要求。
- 如果仓库为空且没有明确引擎，但用户要求生成“场景”和“场景脚本”，默认使用 Godot 4 文本场景骨架。
- 如果请求只适合补单个模块，只创建对应模块的目录、场景和接口。

## 创建目录时

- 先建立父目录，再创建场景、脚本和接口文件。
- 不要把 UI、玩法逻辑和数据定义混在同一层目录里。
- 优先复用仓库中的命名风格；没有现成风格时，让目录名使用 `snake_case`，让场景名和类名使用 `PascalCase`。
- 让目录和文件名能直接映射到需求术语，例如 `inventory`、`battle`、`main_menu`、`reward_popup`。

## 生成 UI 场景时

- 如果创建的是 Godot `.tscn` 场景，先读取并遵守 [Godot TSCN 场景规则](../godot-tscn-ui-rules/SKILL.md)。
- 创建 Godot `.tscn` 场景时，不要依赖容器节点自动布局；优先使用编辑器中可手动调整的显式节点摆放。
- 创建 Godot `.tscn` 场景时，不要通过代码动态创建按钮、图片、文本或其他主要布局元素。
- 图片素材优先使用 `TextureRect` 节点进行占位，方便后续进行素材替换和位置调整。
- 按钮优先使用 `TextureButton` 节点，不要为了省事默认改用普通 `Button`。
- 为每个独立页面、弹窗、HUD 模块生成一个场景文件。
- 只生成最小可用节点树，优先保证结构清晰、节点语义清楚和后续易扩展。
- 让节点名称对应需求里的控件语义，例如 `StartButton`、`CurrencyLabel`、`RewardList`。
- 先复用已有公共组件；没有现成组件时，再生成最小占位节点或场景。
- 不要编造复杂视觉层级；重点是结构、命名和交互入口。
- 生成内容采用utf-8编码，避免乱码

## 生成 UI 脚本时

- 让每个 UI 场景对应一个脚本文件。
- 让脚本只负责缓存节点引用、注册信号、处理界面事件入口和刷新显示状态。
- 不要在 UI 脚本里直接写完整玩法规则或数据存取细节。
- 为明显事件预留处理函数，例如 `_on_start_button_pressed()`、`_on_close_button_pressed()`。
- 如果文档提到初始化、绑定数据或刷新状态，优先留出 `setup()`、`bind_events()`、`refresh_view()` 一类方法。
- 脚本里面的表达式不要使用 := 来赋值
- 在每个方法上方写一句简短注释，只描述职责，利用中文来写
- 生成内容采用utf-8编码，避免乱码

## 生成逻辑接口时

- 按系统拆分接口文件，例如 `inventory_service`、`battle_service`、`quest_service`。
- 让一个文件只表达一个主要职责，不要把无关系统塞进同一接口。
- 只生成方法签名和最小占位体，不提前写具体业务流程。
- 让方法名优先使用动词开头，例如 `load_*`、`start_*`、`update_*`、`complete_*`、`claim_*`。
- 让参数贴合业务实体，不要泛化成 `data`、`info`、`payload` 这类空泛名字，除非文档本身就没有更具体的信息。
- 如果语言支持类型注解，尽量补齐参数和返回类型。
- 脚本里面的表达式不要使用 := 来赋值
- 在每个方法上方写一句简短注释，只描述职责，利用中文来写
- 生成内容采用utf-8编码，避免乱码

## 完成后自查

- 检查目录是否真实创建，而不是只在回复里列出来。
- 检查每个 UI 场景是否都有对应脚本，或者已经明确说明该场景无需脚本。
- 检查场景脚本和逻辑接口是否职责分离，没有把页面控制和游戏规则混在一起。
- 检查方法注释是否简短、准确，并与方法职责一致。
- 检查是否已经为后续实现留出清晰入口，而不是生成无法继续扩展的空壳。
- 检查生成内容采用utf-8编码，避免乱码
- 检查脚本里面的表达式是否使用 := 来赋值
- 检查注释内容是否是中文
