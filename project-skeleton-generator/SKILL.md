# 项目骨架生成

## 概览

使用这个 skill 根据需求分析文档快速落出可继续开发的游戏项目骨架。输出必须包含真实目录、UI 场景、场景脚本和按系统拆分的逻辑接口桩，而不是只给建议、树状图或伪代码清单。

## 工作流程

1. 先读取需求分析文档，提取项目类型、目标平台、引擎线索、核心玩法模块、UI 页面列表和关键系统接口。
2. 再检查当前工作区是否已有项目约定，包括目录布局、命名风格、场景格式、脚本语言和既有分层。
3. 优先继承现有项目约定；只有在工作区为空或缺少明确线索时，才使用 [Godot 4 默认约定](references/godot-defaults.md)。
4. 如果要创建 Godot `.tscn` UI 场景，先读取并遵守 [godot-tscn-ui-rules](../godot-tscn-ui-rules/SKILL.md) 的约束，再开始创建场景节点结构。
5. 先创建真实目录，再创建最小可用文件。不要只输出计划，也不要把所有内容塞进单个目录。
6. 为每个独立 UI 页面、弹窗或 HUD 模块生成对应场景和脚本，让命名能直接映射回需求文档里的界面元素和行为。
7. 为每个核心玩法系统生成独立的逻辑接口文件，只保留接口、抽象方法或服务桩，不提前写具体业务逻辑。
8. 为每个生成的方法补一句简短注释，说明职责，不解释实现细节。
9. 完成后按 [生成规则](references/generation-rules.md) 自查路径、命名、绑定关系和分层是否清晰。

## 生成边界

- 如果仓库已经有稳定结构，沿用它，不要强行改造成另一套模板。
- 如果文档对 UI 细节描述不完整，仍然生成最小场景骨架，并留出可继续扩展的节点和处理函数。
- 如果引擎或框架不明确，但用户明确要求“场景 + 场景脚本”，默认按 Godot 4 处理。
- 如果请求只涉及局部模块，只创建对应目录和文件，不顺手重构整个项目。

## 输出要求

- 真实创建目录和文件，不要只在回复里列树状结构。
- 让每个 UI 场景和对应脚本形成清晰的一对一关系，除非该界面明确不需要单独脚本。
- 如果输出的是 Godot `.tscn` 场景，必须按 [godot-tscn-ui-rules](../godot-tscn-ui-rules/SKILL.md) 创建，不要绕过其中的场景约束。
- 让 UI 脚本只负责界面装配、信号入口和逻辑接口调用，不承载完整玩法规则。
- 让逻辑接口按系统归类，方法命名使用清晰动词，参数和返回值尽量显式。
- 让所有生成的方法都带简短注释，便于后续继续实现。
