# Godot TSCN 场景规则

## 概述

在创建或修改 Godot UI 场景时，优先保证场景结构直观、位置可见、后续易于在编辑器中手动调整。所有界面布局应尽量直接写在 `.tscn` 中，而不是交给容器自动排布或运行时代码生成。

## 硬性规则

- 不要使用基于 `Container` 的节点来搭建主要界面布局。
- 不要默认使用 `HBoxContainer`、`VBoxContainer`、`GridContainer`、`MarginContainer`、`CenterContainer`、`PanelContainer`、`ScrollContainer` 等自动布局容器来组织页面。
- 不要通过 GDScript 动态创建按钮、图片、文字或其他可见布局元素。
- 不要使用 `Button.new()`、`TextureButton.new()`、`Label.new()`、`TextureRect.new()`、`add_child()` 等代码来搭建初始可见界面，除非用户明确要求运行时动态生成 UI。
- 美术资源占位统一使用 `TextureRect`。
- 交互按钮统一优先使用 `TextureButton`，不要默认使用普通 `Button` 来做正式界面按钮。

## 默认搭建方式

- 全屏场景使用 `Control` 作为根节点。
- 弹窗叠加场景使用 `CanvasLayer` 作为根节点。
- 直接在场景树中摆放节点，并明确设置 anchors、offsets、size。
- 所有重要布局节点都应在 `.tscn` 里可见，方便后续手动拖拽和微调。
- 节点命名按用途命名，例如 `BackgroundRect`、`StartButton`、`ConfirmButton`、`PortraitRect`、`RewardFrameRect`。
- 脚本只负责行为、信号绑定、数据刷新和状态同步，不负责搭建界面骨架。

## 节点选型

- 背景图、立绘、边框、美术槽位、卡面插图、装饰图统一使用 `TextureRect`。
- 可点击按钮统一使用 `TextureButton`。
- 文本说明使用 `Label` 或 `RichTextLabel`。
- 如需简单底板，可使用 `Panel`，但不要把它当成自动布局容器来用。
- 如果某个视觉元素会重复出现，优先拆成可复用子场景，再在父场景中手动摆放这些子场景实例。

## 布局要求

- 每个重要节点都要显式设置位置和尺寸。
- 优先采用编辑器可见的手动布局，不要依赖自动分布。
- 间距、对齐和区域划分尽量写在场景本身，而不是放到脚本里计算。
- 如果界面由多个区域组成，优先使用显式放置的节点来表达这些区域，而不是一层层嵌套容器。
- 如果需要列表或重复块，优先考虑预留固定占位节点或复用子场景。
- 只有在用户明确要求“数量动态变化且必须运行时生成”时，才允许改用运行时代码生成重复 UI。

## 脚本要求

- 脚本只负责交互和状态，不负责页面骨架生成。
- 可以在脚本中连接按钮信号，但按钮节点本身必须先在 `.tscn` 中创建好。
- 推荐使用 `setup()`、`bind_events()`、`refresh_view()` 这类方法组织逻辑。
- 不要在 `_ready()` 里动态创建整棵界面树。
- 不要把布局生成逻辑和玩法逻辑混在一起。

## 工作流程

1. 先查看当前项目或现有场景的风格约定。
2. 直接在 `.tscn` 中创建可见的界面节点层级。
3. 所有后续需要替换素材的位置，都先放好 `TextureRect` 占位。
4. 所有按钮都使用 `TextureButton` 创建，并在场景里摆好位置。
5. 再补充 `Label` 等说明控件，并保持显式位置可调整。
6. 在可见结构稳定后，再挂接轻量脚本处理交互。
7. 最后复查场景是否便于在 Godot 编辑器中手动修改。

## 例外情况

- 只有用户明确要求使用容器布局时，才允许使用容器节点。
- 只有用户明确要求运行时动态生成 UI 时，才允许通过代码创建按钮或布局元素。
- 如果正在修改一个已经严重依赖容器的旧场景，默认不要继续扩散该模式；新加内容优先保持手动布局风格，除非用户要求完全沿用旧结构。

## 输出目标

- 产出的 `.tscn` 场景应当结构清晰、可直接在 Godot 编辑器中手调。
- 所有美术替换位应清楚明确。
- 所有按钮应使用 `TextureButton`，而不是为了省事改用普通 `Button`。
- 优先保证“后续好改”，而不是追求自动布局的省事。
