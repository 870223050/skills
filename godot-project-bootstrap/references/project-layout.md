# Godot 项目结构

## 目标

- 让每个功能模块都容易归属、容易删除。
- 让共享层保持精简、稳定、可预期。
- 让 Codex 能只修改一个模块，而不需要重写整个项目。

## 推荐目录树

```text
res://
  autoload/
  core/
  features/
    <feature>/
      scenes/
      scripts/
      ui/
      data/
      tests/
  ui/
    widgets/
    screens/
    themes/
  resources/
    config/
    balancing/
    localization/
  assets/
    art/
    audio/
    fonts/
    vfx/
  tests/
    unit/
    integration/
    smoke/
  addons/
```

## 文件夹规则

### autoload/

这里只放真正全局性的服务，例如场景切换、存档读档、设置、会话状态或事件总线。
不要把具体功能玩法、功能 UI 或重量级场景逻辑放在这里。

### core/

这里放多个功能都会依赖的可复用运行时系统，例如输入抽象、状态机、共享数学工具、存档模型和服务接口。
不要把某个功能专属的场景或一次性玩法规则放在这里。

### features/<feature>/

这里用于承载一个独立玩法切片或一个明确的产品功能区。
把该功能的场景、本地脚本、专属 UI、本地资源和测试都尽量放在一起。
不要直接依赖兄弟功能目录；如果确实需要共享，把可复用部分提取到 `core/` 或 `ui/`。

### ui/

这里放整个项目范围内都可复用的控件、界面和主题。
某个功能专用的 UI 应该保留在对应的功能目录里。

### resources/

这里放非导入型美术资源的数据文件和配置，例如数值表、调优数据和本地化包。
不要把运行时脚本混放到这里。

### assets/

这里只放导入型二进制资源，例如美术、音频、字体、特效等。
不要把玩法逻辑放在这里。

### tests/

这里放 smoke test、集成测试和单元测试。
如果有帮助，可以按生产目录结构镜像组织测试，让每个测试的归属一眼可见。

### addons/

这里放第三方插件或本地插件。
不要修改供应商文件，除非你就是在有意升级或维护这个插件。

## 放置规则

- 新玩法系统：先放进 `features/<feature>/`
- 被 2 个以上功能复用：提取到 `core/` 或 `ui/`
- 全局应用状态或路由：放进 `autoload/`
- 共享表现控件：放进 `ui/`
- 静态数值或本地化数据：放进 `resources/`
- 导入的源美术资源：放进 `assets/`
- 第三方插件：放进 `addons/`

## 命名规则

- 文件夹统一使用小写 `snake_case`
- 场景和脚本名称要清晰，并带有功能归属语义
- 避免使用 `common`、`misc`、`temp` 这类模糊命名，除非内容确实是临时性的
