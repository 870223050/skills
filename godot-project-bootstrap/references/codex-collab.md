# Codex 协作规则

## 目标

让每一次改动都尽量局部、可回退、容易续接。

## 不可妥协的规则

- 尽量只修改必要的最少文件。
- 除非需求明确要求，否则不要重命名或移动共享场景、autoload 或公共 API。
- 功能开发尽量限制在目标 `features/<feature>/` 目录内，除非这次改动明确属于共享层。
- 不要顺手格式化无关文件。
- 当一个小补丁能解决问题时，不要用大范围重构替代它。

## 安全改动顺序

1. 先确认归属层。
2. 阅读现有入口点和公共接口。
3. 做最小且尽量保持行为不变的修改。
4. 添加或更新 smoke test、启动检查或最小验证。
5. 只有当同样模式在多个功能里重复出现时，才提取共享代码。

## 当接口必须变化时

- 先增加适配层或兼容包装。
- 在调用方完成迁移之前，保留旧调用路径可用。
- 如果改动会跨越模块边界，先说明影响范围，再开始编辑。

## 推荐模式

- 以功能目录为中心的本地实现，通过信号集成。
- 组合优先于继承。
- 在 `core/` 和 `autoload/` 中保持共享服务轻量。
- 在抽象之前，优先采用基于场景的纵向切片。
- 只有在调用方已更新后，才做保持兼容的重命名。

## 需要暂停并重新收敛范围的情况

如果改动会导致以下情况，请先暂停并重新界定范围：

- 触及 autoload 注册
- 跨多个归属层移动文件
- 重命名已经被其他代码使用的公共场景、脚本、资源或信号
- 需要跨多个无关功能同时修改

## 自查清单

- 这次改动能否限制在一个功能目录里完成？
- 在新增路径存在的同时，旧调用方能否继续工作？
- 共享层是否比功能层更小、更稳定？
- 我们是否用启动检查或 smoke test 验证了这次改动？
